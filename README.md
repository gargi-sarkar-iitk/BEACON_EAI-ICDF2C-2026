This repository contains the official code, datasets, and evaluation scripts for the paper:
> **BEACON: A Unified Behavioral-Tactical Framework for Explainable Cybercrime Analysis with Large Language Models**  
> *EAI ICDF2C 2026 — International Conference on Digital Forensics & Cyber Crime*
---
Overview
BEACON is a dual-axis classification framework that analyzes natural-language cybercrime narratives (victim reports, incident stories) and produces structured, explainable assessments across 20 labels spanning two complementary dimensions:
Tactical Axis (14 labels) — Attack stages adapted from adversarial kill-chain methodology: `reconnaissance`, `resource_development`, `initial_contact`, `detonation`, `persistence`, `escalation`, `defense_evasion`, `credential_harvesting`, `discovery`, `pivoting`, `collection`, `command_and_control`, `exfiltration`, `impact`.
Behavioral Axis (6 labels) — Psychological manipulation theories grounded in social engineering research: `fear_and_intimidation`, `phantom_riches`, `authority_socialproof_impersonation`, `urgency_and_scarcity`, `consistency_and_reciprocity`, `emotional_exploitation`.
For each label the model outputs a binary classification (`Present: Yes/No`) alongside a natural-language rationale grounded exclusively in the input story — making every prediction fully explainable.
The system is built by fine-tuning Mistral-7B-Instruct-v0.2 with LoRA adapters (via PEFT + TRL) on a curated dataset of 1,500 labeled cybercrime narratives.

│   ├── train_script.py               # QLoRA fine-tuning using TRL/SFTTrainer

Dataset
`dataset/cybercrime_dataset_1500.csv` contains 1,500 first-person cybercrime narratives covering a wide range of fraud types (phishing, romance scams, sextortion, tech-support fraud, investment fraud, etc.). Each row includes:
Column	Description
`Story`	The raw narrative text
`Fraud_Type`	High-level fraud category
`Major_Tactic`	Primary attack stage
`Major_Theory`	Primary psychological theory
`<Label>`	Binary label for each of the 20 BEACON dimensions
`<Label>_Reason`	Ground-truth natural-language rationale
`dataset/testset.csv` follows the same schema and is used exclusively for evaluation.

Requirements
Python 3.8+
CUDA-capable GPU (≥24 GB VRAM recommended; the scripts use 4-bit quantization via `bitsandbytes`)
Install dependencies:
```bash
python3 -m venv venv
source venv/bin/activate   # On Windows: venv\Scripts\activate
pip install torch transformers peft trl bitsandbytes datasets \
            pandas tqdm scikit-learn evaluate
```
> **Note on `bitsandbytes`:** Ensure your `bitsandbytes` version is compatible with your CUDA version. See the [bitsandbytes installation guide](https://github.com/TimDettmers/bitsandbytes) if you encounter issues.

Quickstart
Inference
Run the analyzer on a sample story using either the base Mistral model or the fine-tuned BEACON adapters:
```bash
# Fine-tuned model (default adapter path: ./results_scam_analyzer_stable/final_model)
python scripts/inference_script.py --model_type finetuned

# Base model only (no adapters, uses the detailed definitional prompt)
python scripts/inference_script.py --model_type base

# Fine-tuned model with a custom adapter path
python scripts/inference_script.py --model_type finetuned \
    --adapter_path ./results_scam_analyzer_stable/checkpoint-282
```
The script ships with a built-in demo story (a sextortion scenario). To analyze your own stories, import and call `analyze_story()` directly:
```python
from scripts.inference_script import load_model_and_tokenizer, analyze_story

model, tokenizer = load_model_and_tokenizer("finetuned", adapter_path="./results_scam_analyzer_stable/final_model")
result = analyze_story("Your cybercrime narrative here...", model, tokenizer, "finetuned")
print(result)
```
Example output format:
```
[reconnaissance]
Present: Yes
Reason: The scammer viewed the victim's design portfolio on Instagram before making contact.

[initial_contact]
Present: Yes
Reason: The scammer messaged the victim through Instagram posing as a talent scout.

[fear_and_intimidation]
Present: Yes
Reason: The attacker threatened to distribute fabricated explicit material to the victim's contacts.
...
```

Training
Fine-tuning uses LoRA (Low-Rank Adaptation) applied to all projection layers of Mistral-7B-Instruct-v0.2 via `peft` and `trl.SFTTrainer`.
```bash
python scripts/train_script.py
```
Key training configuration (see `Config` class in `train_script.py`):
Parameter	Value
Base model	`mistralai/Mistral-7B-Instruct-v0.2`
LoRA rank (`r`)	32
LoRA alpha	64
LoRA dropout	0.05
LoRA target modules	`q_proj`, `k_proj`, `v_proj`, `o_proj`, `gate_proj`, `up_proj`, `down_proj`
Quantization	4-bit NF4 (bitsandbytes)
Epochs	3
Learning rate	5e-5
LR scheduler	Cosine
Optimizer	`paged_adamw_32bit`
Max sequence length	4096
Batch size	8 (+ 2 gradient accumulation steps)
Adapters and tokenizer are saved to `results_scam_analyzer_stable/final_model` after training completes.

Evaluation
The evaluation script benchmarks both the base and fine-tuned models on the held-out test set, computing:
Classification metrics — per-label precision, recall, F1, and accuracy via `scikit-learn`
Text quality metrics — ROUGE, BLEU, and BERTScore for the generated rationale text
LaTeX output — a formatted comparison table written to `evaluation_results.tex`
```bash
python scripts/evaluation_script.py \
    --adapter_path ./results_scam_analyzer_stable/final_model \
    --data_path dataset/testset.csv \
    --num_samples 100        # Use -1 to evaluate on the full test set
```
> **Parsing note:** The evaluation parser expects the structured `[label]\nPresent: Yes/No\nReason: ...` output format. If you modify the prompt templates in `inference_script.py`, update the parser in `evaluation_script.py` accordingly.
---
Prompt Strategy
BEACON uses two distinct prompt strategies depending on which model variant is used:
Model	Prompt Type	Description
Base Mistral-7B	Detailed definitional prompt	Full definitions for all 20 labels are included in-context to guide zero-shot classification
Fine-tuned BEACON	Concise task prompt	A short instruction prompt; the model relies on fine-tuned weights for label semantics
This design was specifically chosen to produce a fair base-vs-fine-tuned comparison: the base model is given maximum definitional guidance, while the fine-tuned model demonstrates internalized knowledge of the BEACON taxonomy.
